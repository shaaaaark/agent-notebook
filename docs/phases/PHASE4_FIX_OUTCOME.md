# Phase 4 预修复成果文档

## 1. 阶段目标

本阶段不是继续盲目推进 Phase 5，而是先处理会污染 Phase 4 真实评估结论的基础问题：

- 上传链路前后端不一致
- hybrid 检索低信心判定逻辑不兼容
- fallback 检索链路在当前运行态下会被过短 timeout 误伤

这样做的目的，是先把系统恢复到“可验证状态”，避免后续所有评估结论失真。

## 2. 背景问题

进入本阶段前，项目已有混合检索、BM25、RRF、rerank 等能力，但存在两个非常危险的问题：

1. **前端支持多文件上传，后端却只收单文件**
   - 用户以为上传了多个文件
   - 实际知识库可能只收进一个文件
   - 导致检索、评估、调参结果全部失真

2. **Phase 4 使用了不合适的低信心判定方式**
   - 原逻辑把 hybrid 的分数也按纯向量阈值处理
   - 极容易导致系统把本来有召回信号的结果误判成 clarify

## 3. 实施内容

### 3.1 功能改动

- 后端 ingest 改为真正支持多文件上传
- RAG 低信心判定改为按检索策略分别处理

### 3.2 代码改动

#### `server/src/modules/ingest/ingest.controller.ts`
- `FileInterceptor` → `FilesInterceptor`
- `@UploadedFile()` → `@UploadedFiles()`
- 返回值从单文件结果改为批量结果

#### `server/src/modules/rag/rag.service.ts`
- 增加 `hasEnoughSignal()`
- hybrid 场景下不再直接用统一阈值否决所有结果
- `lowConfidence` 改为按策略判断

### 3.3 配置 / 文档改动

- 新增 `EXECUTION_PLAN.md`，正式把 roadmap 执行顺序重排为“先修基础可信度，再做能力增强”
- 新增阶段成果模板与 Stage A 文档

## 4. 关键设计说明

### 4.1 为什么这样改

这次改动不是为了“加功能”，而是为了恢复工程判断的可信度。

- 上传链路不一致，会污染知识库输入
- 低信心判定错误，会污染 retrieval / eval 输出

这两个问题不修，后面做 policy、gate、replay 都没有意义。

### 4.2 方案取舍

#### 上传链路
- 方案 A：前端降级为单文件
- 方案 B：后端升级为多文件

本次选择 **方案 B**，因为前端已有多文件交互，保留能力更符合知识库产品的预期。

#### 低信心判定
- 方案 A：直接降低阈值
- 方案 B：按检索策略区分判定语义

本次选择 **方案 B**，因为真正的问题不是阈值高低，而是不同分数体系不能混用。

### 4.3 已知限制

- 还未完成真实回归复测
- rerank/embedding 运行态是否仍有额外问题，需下一步确认
- 当前本地 fallback 语义下，系统仍出现 `clarify_rate = 1.0`，说明还需要继续校正信号阈值或检索策略

## 5. 验收结果摘要

- 构建：通过
- 接口一致性：已修复
- 低信心判定根因：已修复一项高概率问题
- 运行态超时误伤：已修复
- 回归结果：22 篇语料已导入；使用 `eval/harness.ts --host http://127.0.0.1:8788` 跑 30 case，当前 `clarify_rate = 1.0`，说明 Stage A 尚未完全恢复基线

## 6. 对后续阶段的影响

本阶段完成后，下一步可以更可信地做：

- 真正的 Phase 4 回归复测
- 参数调优
- policy / thresholds / replay

也就是说，这一阶段的价值不在“新增功能”，而在于：

> **恢复工程链路的可信度，让后续所有评估结果更值得相信。**
